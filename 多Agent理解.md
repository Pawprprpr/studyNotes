# 多Agent系统核心知识点总结

## 一、为什么需要多Agent

单Agent在处理复杂任务时的局限：

- **上下文过长**：一个Agent需同时处理多种异质任务，容易混乱
- **专业知识冲突**：不同任务需要不同领域知识，单个Agent难以全面覆盖
- **无法并行**：没有分工就无法并行执行，效率低下

**多Agent的本质**：将单体Agent拆分为专业集群，每个Agent专注一件事，通过协作完成复杂任务。

## 二、多Agent协作的三种核心模式

### 模式1：顺序编排 (Pipeline / Sequential)

| 项目 | 说明 |
| :--- | :--- |
| **核心机制** | 流程固定，按预定义顺序执行 |
| **决策方式** | 静态，运行前已确定执行顺序 |
| **通俗类比** | 流水线：A做完→B→C |
| **适用场景** | 任务步骤清晰、固定的工作流 |
| **代表框架** | `CrewAI` 的 `Process.sequential` |

### 模式2：集中调度 (Router / Supervisor)

| 项目 | 说明 |
| :--- | :--- |
| **核心机制** | 一个总Agent将子Agent视为工具，自行判断调用谁 |
| **决策方式** | 动态，由总Agent运行时决策 |
| **通俗类比** | 调度中心：总指挥根据任务分发给不同专家 |
| **适用场景** | 任务多样、需动态分配，子Agent数量3-5个最佳 |
| **代表框架** | `LangChain`/`LangGraph` 的 `Supervisor` 模式 |

### 模式3：动态移交 (Handoff / Swarm)

| 项目 | 说明 |
| :--- | :--- |
| **核心机制** | Agent之间平等，处理不了时主动转交给其他Agent |
| **决策方式** | 动态，由当前Agent自行判断是否移交 |
| **通俗类比** | 责任链/甩锅：A说"我干不了，你找B" |
| **适用场景** | 用户意图跳变、跨领域交叉协作 |
| **代表框架** | `OpenAI Swarm` 的 `Handoff` 模式 |

---

## 三、三种模式对比

| 维度 | 顺序编排 | 集中调度 | 动态移交 |
| :--- | :--- | :--- | :--- |
| **决策者** | 预定义规则 | 总Agent | 每个Agent自己 |
| **流程确定性** | 高 | 中 | 低 |
| **灵活性** | 低 | 中 | 高 |
| **Agent数量适应** | 不限 | 3-5个最佳 | 不限 |
| **典型设计模式** | 责任链 | 策略+工厂 | 责任链(动态版) |

---

## 四、层级Agent (Hierarchical Agent)

- **定义**：总Agent调子Agent，子Agent还可再调孙Agent，形成嵌套结构
- **"无限套娃"可行但受限**：工程实践中通常不超过3层，否则延时和错误传递会失控
- **扩展策略**：子Agent过多时，按领域拆分为多个Supervisor，各管一摊
